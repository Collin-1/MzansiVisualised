# Dashboard Section Deep Dive

This document explains the dashboard section of the app in detail, with focus on the interactive Province House Prices dashboard and how it is embedded into the issue page shell.

## 1. What the dashboard section is

In this project, the dashboard section is the interactive data panel shown on the issue detail route:

- Route: `/issues/province-house-prices`
- Host page: `app/issues/[slug]/page.tsx`
- Dashboard component: `components/visualizations/ProvinceHousePrices.tsx`

The issue page acts as the editorial container, while the dashboard component is the actual analytical UI.

## 2. How the dashboard is loaded

The dashboard is resolved dynamically through a visualization registry.

### Registry flow

1. Issue metadata includes a `component` name.
2. `app/issues/[slug]/page.tsx` reads the issue by slug.
3. The page looks up the matching component in `VIZ_REGISTRY`.
4. The selected component is rendered inside the visualization section.

This keeps the issue framework generic and allows multiple dashboard types to be plugged in later.

## 3. Dashboard structure (Province House Prices)

The `ProvinceHousePrices` dashboard is composed of four major zones:

1. Stat bar (top summary)
2. Main map canvas (left)
3. Ranked list sidebar (right)
4. Detail panel (bottom)

### 3.1 Stat bar

The top row contains four summary tiles:

- Most expensive
- Best value
- National median
- Western Cape vs Free State gap

These are presented as quick headline metrics to set context before interaction.

### 3.2 Map area

The map renders all nine provinces as SVG paths generated by D3.

Key points:

- SVG size is fixed in logical units (`W=620`, `H=480`) and scales responsively.
- `geoMercator().fitSize([W, H], SA_GEOJSON)` is used to fit geometry into the viewport.
- `geoPath(projection)` converts geometry into path data (`d`) for each province.

### 3.3 Ranked sidebar

A vertical ranking list shows all provinces sorted by average price (`PROVINCES_BY_RANK`).

Each row includes:

- Rank number
- Color strip (matches map fill)
- Province name
- Average price
- Relative mini-bar based on the maximum price

Hovering a row highlights that province in the map. Clicking sets it as selected.

### 3.4 Detail panel

Below the map, the selected or hovered province shows richer context:

- Province name
- Description
- Average price
- YoY growth
- Capital
- Difference vs national median

This creates a two-level insight model: quick scan in the map, deeper read in the detail panel.

## 4. Data sources and transformations

The dashboard uses two core data inputs:

1. Tabular province metrics
2. Geographic boundaries

### 4.1 Metrics data

From: `data/province-house-prices.ts`

Provides:

- `PROVINCE_DATA` keyed by province code (WC, GP, etc.)
- `PROVINCES_BY_RANK` sorted list for sidebar display
- `NATIONAL_MEDIAN` reference value

### 4.2 Boundary data

From: `publicdata/sa-provinces-boundaries.json`

Each feature includes properties such as `shapeName` and geometry (`Polygon` or `MultiPolygon`).

### 4.3 Geometry rewinding fix

A critical preprocessing step is applied in `rewindGeometryForD3(...)`.

Why it exists:

- Some boundary datasets have ring winding orientation that can cause D3 to treat polygons incorrectly.
- When this happens, `geoBounds` behavior can degrade and map fitting can collapse into tiny output.

What the code does:

- Reverses polygon rings for both `Polygon` and `MultiPolygon` features before projection.

Result:

- Correct South Africa province rendering at the expected scale.

### 4.4 Province name normalization and mapping

`normalizeProvinceName(...)` strips non-letters and lowercases names.

Then `NAME_TO_CODE` maps normalized boundary names to internal province codes.

This makes the dashboard robust to minor naming differences in source data.

## 5. Interaction model

The dashboard uses three state values:

- `selected: ProvinceCode`
- `hovered: ProvinceCode | null`
- `tooltip: { x, y, code } | null`

### 5.1 Active province rule

`activeCode = hovered ?? selected`

Meaning:

- Hover temporarily takes priority.
- Selection remains persistent when hover ends.

### 5.2 Mouse interactions

On map path:

- `onMouseEnter`: sets hovered province and opens tooltip
- `onMouseMove`: updates tooltip position relative to SVG bounds
- `onMouseLeave`: clears hover and tooltip
- `onClick`: sets selected province

On sidebar row:

- `onMouseEnter`: sets hovered province
- `onMouseLeave`: clears hover
- `onClick`: sets selected province

This creates synchronized two-way highlighting between map and list.

### 5.3 Tooltip behavior

Tooltip is absolutely positioned inside the map container and follows cursor movement.

An edge-aware transform shifts the tooltip left when near the right side of the canvas to reduce clipping.

## 6. Visual system and styling

The dashboard sits inside the app's premium surface design language.

### 6.1 Global style tokens

From `app/globals.css`:

- Brand variables (`--gold`, `--earth`, `--bark`, etc.)
- UI surface variables (`--ui-panel`, `--ui-line`, `--ui-shadow`)
- Utility classes (`page-shell`, `ui-card`, `btn-primary`, `btn-secondary`)

### 6.2 Dashboard-specific classes

Map paths use:

- `.province-path`
- `.province-path.is-active`
- `.province-path.is-dimmed`

These provide hover brightness and de-emphasis behavior for non-hovered shapes.

## 7. Host page responsibilities

The issue page (`app/issues/[slug]/page.tsx`) handles:

- Metadata generation for SEO and social cards
- Breadcrumb and issue header
- Rendering the dashboard section
- Narrative analysis below the dashboard
- Share links and navigation back to all issues

So the dashboard is both a data UI and part of a larger editorial storytelling flow.

## 8. Performance notes

Current strengths:

- Dashboard is client-side only where needed (`use client` in visualization file)
- Issue page itself remains mostly server-rendered
- Static issue pages are generated at build time via `generateStaticParams`

Potential enhancements:

- Memoize converted SVG paths if geometry grows larger
- Lazy-load heavier future visualizations similarly to current registry approach
- Move static top metrics from hardcoded values to computed values from source data

## 9. Known implementation trade-offs

1. `fitSize` currently uses a narrow type cast for strict TypeScript compatibility with D3 geo type definitions.
2. Top stat bar values are presently hardcoded strings.
3. The map component still includes some dark-mode classes even though the new visual system is primarily light-theme forward.

These do not block functionality but are good candidates for cleanup.

## 10. How to extend this dashboard

If you want to evolve this section, the clean extension points are:

1. Data updates: modify `data/province-house-prices.ts`
2. Geometry updates: replace `publicdata/sa-provinces-boundaries.json`
3. New interaction mode: add filters or legends inside `ProvinceHousePrices.tsx`
4. New dashboard issue: create a new visualization component and register it in `components/visualizations/index.tsx`

That keeps the dashboard architecture scalable while preserving the issue-driven publishing model.
